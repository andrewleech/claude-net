# claude-net Specification

> **2026-04 update (dashboard restructure, Phase 1):** mirror-session is now always on and trusts the hub's network boundary (LAN / Tailscale / reverse proxy with auth) — the per-session owner/reader token model, the `/mirror/:sid` permalink, and the `mirror_on` / `mirror_off` / `mirror_status` / `mirror_url` / `mirror_share` / `mirror_revoke` / `mirror_consent` MCP tools have all been removed. The sections below still reference the token model in places; treat those as historical context.

## Overview

claude-net is a lightweight messaging hub that enables Claude Code agents on a LAN to communicate through named identities. A single Docker container runs the hub (Bun + Elysia), which routes messages between agents, serves a built-in monitoring dashboard, and hosts the plugin script that Claude Code fetches at startup.

No API keys, no Redis, no external dependencies. Network-level isolation provides the trust boundary.

## Goals and Objectives

- Zero-config agent communication: agents auto-register with a default name derived from their working directory and hostname
- Natural language interaction: agents register and send messages through conversational prompts ("send this report to the reviewer agent")
- Single-container deployment: one `docker run` command starts the entire system
- No local files to manage: the plugin script is served by the hub and fetched by Bun at each session start

## Target Users

Developers running multiple Claude Code sessions concurrently who want those sessions to coordinate — delegate tasks, share results, broadcast status, or collaborate in named teams.

## Functional Requirements

### FR-1: Agent Identity

**FR-1.1: Default identity**
On startup, the plugin auto-generates an identity in the format `session:user@host` where session is `basename(cwd)`, user is the OS username, and host is the hostname. For example, a session in `/home/andrew/myproject` on a machine named `laptop` becomes `myproject:andrew@laptop`.

**FR-1.2: Custom identity**
The `register` MCP tool allows an agent to override the default name. A bare name like `reviewer` is auto-expanded to `reviewer:user@host`.

**FR-1.3: Name uniqueness**
Names are globally unique. The first registration claims a name. The same agent (same WebSocket connection context) can re-register with the same name (reconnect semantics). A different agent attempting the same name receives an error with the message: "Name 'X' is already registered. Choose a different name."

**FR-1.4: Addressing**
Agents can be addressed by:
- Full name: `myproject:andrew@laptop` — exact match
- Session + user: `myproject:andrew` — matches across hosts
- User + host: `andrew@laptop` — matches across sessions
- Plain string: tries session name, then user name, then host name. If ambiguous, the hub returns an error listing the full names.

**FR-1.5: No persistence**
Agent names are ephemeral. No settings file is written. On session end, the name is released (subject to team membership timeout per FR-4.4).

### FR-2: Messaging

**FR-2.1: Message types**
Two message types:
- `message` — a standalone message
- `reply` — a response to a previous message, carrying a `reply_to` field

**FR-2.2: Message fields**
Every message has:
- `message_id` — UUID generated by the hub on receipt
- `from` — sender's full `name@host` identity (stamped by hub, not client-supplied)
- `to` — recipient name, `broadcast`, or `team:teamname`
- `type` — `message` or `reply`
- `content` — string body
- `reply_to` — (optional) `message_id` of the message being replied to. Required when `type` is `reply`.
- `timestamp` — ISO 8601, set by hub

**FR-2.3: Direct messaging**
`send_message(to, content, reply_to?)` sends to a single agent by name. If the recipient is not connected, the tool returns an error: "Agent 'X' is not online."

**FR-2.4: Broadcast**
`broadcast(content)` sends to all connected agents except the sender. Returns the count of agents the message was delivered to.

**FR-2.5: Team messaging**
`send_team(team, content, reply_to?)` sends to all online members of the named team, excluding the sender if they are a member. If the team doesn't exist, returns an error. If no members are online, returns an error.

**FR-2.6: No queuing**
Messages are not queued. If the recipient is offline, the message is not stored. The sender is informed that delivery failed.

### FR-3: Channel Tag Format

Inbound messages are delivered to Claude Code as `notifications/claude/channel` MCP notifications. They render as `<channel>` tags in the session context.

**FR-3.1: Tag attributes**
- `source` — always `"claude-net"` (set automatically by Claude Code from the MCP server name)
- `from` — sender's full `name@host` identity
- `type` — `message` or `reply`
- `message_id` — the message's UUID
- `reply_to` — (present only for replies) the `message_id` being replied to
- `team` — (present only for team messages) the team name the message was sent to

**FR-3.2: Examples**

Direct message:
```xml
<channel source="claude-net" from="review:andrew@laptop" type="message" message_id="abc123">
  Here's the code review you asked for...
</channel>
```

Reply:
```xml
<channel source="claude-net" from="project:bob@desktop" type="reply" message_id="def456" reply_to="abc123">
  Fixed the issues you found.
</channel>
```

Team message:
```xml
<channel source="claude-net" from="project:bob@desktop" type="message" message_id="ghi789" team="backend">
  Everyone on backend team: avoid modifying src/auth/ for the next hour.
</channel>
```

### FR-4: Teams

**FR-4.1: Implicit creation**
Teams are created when the first agent joins. No explicit create command.

**FR-4.2: Implicit deletion**
Teams are deleted when the last member leaves (or their membership times out).

**FR-4.3: Join and leave**
- `join_team(team)` — adds the agent to the team. If the team doesn't exist, it is created.
- `leave_team(team)` — removes the agent from the team. If the team becomes empty, it is deleted.

**FR-4.4: Membership timeout on disconnect**
When an agent disconnects, their team memberships are preserved for 2 hours. If the agent reconnects (with the same name) within that window, memberships are restored. After 2 hours, memberships are removed and the agent is treated as having left all teams.

**FR-4.5: Team listing**
`list_teams()` returns all teams with their member names and online/offline status of each member.

### FR-5: Discovery

**FR-5.1: List agents**
`list_agents()` returns all connected agents with:
- `name` — full `name@host` identity
- `status` — `online` or `offline` (offline only appears during the 2h team membership window)
- `teams` — list of team names the agent belongs to

**FR-5.2: List teams**
`list_teams()` returns all teams with:
- `name` — team name
- `members` — list of member names with online/offline status

### FR-6: MCP Tool Surface

The plugin exposes these MCP tools:

| Tool | Parameters | Returns | Description |
|------|-----------|---------|-------------|
| `whoami` | _(none)_ | `{name}` or error | Return current registered name |
| `register` | `name: string` | `{name, full_name}` or error | Override default identity (bare name auto-expands to session:user@host) |
| `send_message` | `to: string, content: string, reply_to?: string` | `{message_id, delivered: true}` or error | Send to agent by name (supports all addressing modes) |
| `broadcast` | `content: string` | `{message_id, delivered_to: number}` | Send to all online agents |
| `send_team` | `team: string, content: string, reply_to?: string` | `{message_id, delivered_to: number}` or error | Send to team members |
| `join_team` | `team: string` | `{team, members: string[]}` | Join a team (creates if new) |
| `leave_team` | `team: string` | `{team, remaining_members: number}` | Leave a team |
| `list_agents` | _(none)_ | `Agent[]` | List all agents |
| `list_teams` | _(none)_ | `Team[]` | List all teams with members |
| `ping` | _(none)_ | `{pong: true}` | Test channel round-trip (hub echoes back as channel notification) |

### FR-7: MCP Server Instructions

The plugin's `instructions` string (injected into Claude's system prompt) covers:
- How inbound messages appear (`<channel>` tag format and attributes)
- How to send messages (tool names and when to use each)
- That `from` is always full `name@host`
- That messages to offline agents fail (no queuing)
- That `reply_to` should be set when responding to a specific message

### FR-8: Mirror Sessions (stub — detailed behavior lands with phases M1–M4)

Mirror sessions let a browser on the trust network follow along with a local Claude Code session and (with an owner token) inject new user prompts back into it. Capture is via Claude Code hooks, transported through a local mirror-agent daemon to the hub, and rendered on a new per-session web view.

**FR-8.1: Event kinds**

Each event emitted for a mirror session has one of these kinds:
- `session_start`, `session_end`
- `user_prompt`, `assistant_message`
- `tool_call`, `tool_result`
- `notification`, `compact`

**FR-8.2: Plugin → Hub frame (outbound mirror events)**

```typescript
{
  action: "mirror_event",
  sid: string,               // Claude Code session UUID
  uuid: string,              // event UUID (for dedupe with JSONL reconciliation)
  kind: MirrorEventKind,
  ts: number,                // epoch ms
  payload: MirrorEventPayload,  // kind-discriminated
  requestId?: string
}
```

**FR-8.3: Hub → mirror-agent frames (inbound control)**

```typescript
// Inject a user prompt into the live local session (Phase M2+).
// Hub sends these on /ws/mirror/:sid to the connected agent-role client.
{
  event: "mirror_inject",
  sid: string,
  text: string,
  seq: number,                 // monotonic per session, starting at 1
  origin: { watcher: string, ts: number }
}

// Pause / resume / close the mirror session from the hub side
{
  event: "mirror_control",
  sid: string,
  op: "pause" | "resume" | "close"
}
```

**FR-8.3.1: Inject REST endpoint**

Dashboard clients submit injects through the hub:

```
POST /api/mirror/:sid/inject?t=<owner-token>
Body: { "text": string, "watcher"?: string }
→ 200 { accepted: true, seq: number }   on success
→ 400                                    empty / missing text
→ 401                                    missing token
→ 403                                    bad token / reader token used
→ 413                                    prompt > 32 KB
→ 429                                    rate limit (≥ 1 per 250ms per session)
→ 503                                    mirror-agent not connected
```

Reader-type tokens (FR-8.5) cannot inject — the endpoint returns 403.

**FR-8.4: Dashboard events**

The dashboard `/ws/dashboard` channel gains these events:
- `mirror:session_started`, `mirror:session_ended`
- `mirror:event` (delivered to per-`sid` subscribers on `/ws/mirror/{sid}`)
- `mirror:watcher_joined`, `mirror:watcher_left`

**FR-8.5: Tokens**

Each mirror session carries one or more tokens. Each token is `{ value: string (128-bit hex), type: "owner" | "reader", sid, created_at, revoked_at? }`. Tokens are delivered via the URL fragment (`#token=...`) so they are never sent in HTTP `Referer` or landed in hub access logs. Owner tokens authorize read, inject, share, revoke, and close. Reader tokens authorize read only.

**FR-8.6: Public summary model**

```typescript
// GET /api/mirror/sessions (owner-gated)
{
  sid: string,
  owner_agent: string,
  cwd: string,
  created_at: string,
  last_event_at: string,
  watcher_count: number,
  transcript_len: number
}
```

**FR-8.6.1: Share / revoke (Phase M3+)**

Owners can mint additional tokens and revoke any token they own:

```
POST /api/mirror/:sid/share?t=<owner>
→ 200 { reader_token, mirror_url }
→ 403                                   not an owner token

POST /api/mirror/:sid/revoke?t=<owner>
Body: { "token"?: string, "all"?: boolean }
→ 200 { revoked: number }
→ 403                                   not an owner token
→ 404                                   token value not found

GET /api/mirror/:sid/tokens?t=<owner>
→ 200 { tokens: [ { type, token_preview, created_at, revoked_at } ] }
→ 403                                   not an owner token
```

Revoking a token immediately closes any live WebSocket the holder had open. `all:true` revokes every token on the session.

**FR-8.6.2: Redactor (Phase M3+)**

The mirror-agent runs every event through a regex redactor before shipping it to the hub. Defaults cover AWS access keys, GitHub PATs, Anthropic / OpenAI key prefixes, PEM headers, and JWTs. User-extensible via `~/.claude-net/redact.json` (merged with project-local `.claude-net/redact.json`). Redaction is best-effort — documented as a convenience, not a compliance control.

**FR-8.6.3: Archive (Phase M3+)**

When `CLAUDE_NET_MIRROR_STORE=<dir>` is set, the hub persists every mirror session as `<dir>/<sid>.jsonl`. Post-restart, transcripts are reachable at:

```
GET /api/mirror/archive/:sid
→ 200 { sid, owner_agent, cwd, created_at, closed_at, transcript[] }
→ 404                                   not archived
```

Archive retention defaults to 24 hours; override with `CLAUDE_NET_MIRROR_RETENTION_HOURS`. Live sessions are NOT auto-restored — archive is post-mortem viewing only.

**FR-8.6.4: TLS (Phase M3+)**

Set both `CLAUDE_NET_TLS_CERT` and `CLAUDE_NET_TLS_KEY` to file paths and the hub binds HTTPS/WSS on its normal port. Generated mirror URLs switch to `https://` / `wss://`.

**FR-8.6.5: Rate limits (Phase M3+)**

- `POST /api/mirror/session` — 30 per 5 minutes per remote IP.
- `POST /api/mirror/:sid/inject` — one per 250ms burst ceiling, plus `CLAUDE_NET_MIRROR_INJECT_RPM` (default 20) per minute.

Breached requests return `429` with `Retry-After` in seconds.

**FR-8.7: Remote-inject consent (Phase M2+)**

The mirror-agent gates every remote inject through a per-session consent policy. claude-net is designed for private-trust-network use (tailnet / LAN), so the default accepts every inject without prompting. Two modes:

- `always` — default; accept every inject.
- `never` — reject every inject (effectively makes the mirror read-only for that session).

The policy is set via the plugin's `mirror_consent` tool (with mode `reset` to revert to the default). Legacy mode names (`ask-first-per-session`, `ask-every-time`) are accepted for backward compatibility and coerced to `always`.

Successful injects are logged back into the transcript as a `notification` event (`source: "mirror-agent"`) for auditability. Rejections / failures are also logged, so watchers can see outcomes.

Phase M1 implements read-only outbound mirror and the owner token. Phase M2 adds tmux-based injection + consent. Phase M3 adds reader tokens, a redactor, optional disk persistence, optional TLS, and stricter rate limiting. Phase M4 (opt-in) replaces tmux injection with a same-length binary patch. Full per-phase spec lives in `docs/MIRROR_SESSION_IMPLEMENTATION_PLAN.md` and `docs/MIRROR_SESSION_PHASE_{0..4}.md`.

## Architecture

### Hub Server

Single Bun process running Elysia. Serves:
- **WebSocket endpoint** (`/ws`) — plugin connections for message routing
- **WebSocket endpoint** (`/ws/dashboard`) — dashboard connections for live updates
- **REST API** (`/api/*`) — agent listing, team listing, message sending from dashboard
- **Static files** — dashboard HTML at `/`, plugin script at `/plugin.ts`
- **Setup endpoint** (`/setup`) — shell script for MCP registration

### Plugin

Single TypeScript file (`plugin.ts`) served by the hub. Claude Code spawns it as a stdio subprocess via `bun run http://hub:4815/plugin.ts`. The plugin:
1. Creates an MCP Server with `claude/channel` capability and `tools` capability
2. Connects to Claude Code over stdio
3. Opens a WebSocket to the hub using `CLAUDE_NET_HUB` env var
4. Auto-registers with default name `session:user@host`
5. Exposes 10 MCP tools
6. Forwards inbound hub messages as `notifications/claude/channel` notifications

### WebSocket Protocol

**Plugin → Hub frames:**

All outbound frames include an optional `requestId` string. The hub echoes it in the response frame when present. Operations that need confirmation (register, send_message, join_team, etc.) should include a `requestId`. Broadcasts and other fire-and-forget operations may omit it.

```typescript
// Register / override name
{ action: "register", name: string, requestId?: string }

// Send direct message
{ action: "send", to: string, content: string, type: "message" | "reply", reply_to?: string, requestId?: string }

// Broadcast
{ action: "broadcast", content: string, requestId?: string }

// Send to team
{ action: "send_team", team: string, content: string, type: "message" | "reply", reply_to?: string, requestId?: string }

// Join team
{ action: "join_team", team: string, requestId?: string }

// Leave team
{ action: "leave_team", team: string, requestId?: string }

// List agents
{ action: "list_agents", requestId?: string }

// List teams
{ action: "list_teams", requestId?: string }
```

**Hub → Plugin frames:**

```typescript
// Response to a request
{ event: "response", requestId: string, ok: boolean, data?: any, error?: string }

// Inbound message (unsolicited push)
{ event: "message", message_id: string, from: string, to: string, type: "message" | "reply", content: string, reply_to?: string, team?: string, timestamp: string }

// Registration confirmed (unsolicited, on initial auto-register)
{ event: "registered", name: string, full_name: string }

// Error (unsolicited)
{ event: "error", message: string }
```

**Hub → Dashboard frames:**

```typescript
// Agent connected
{ event: "agent:connected", name: string, full_name: string }

// Agent disconnected
{ event: "agent:disconnected", name: string, full_name: string }

// Message routed (all messages, for the feed)
{ event: "message:routed", message_id: string, from: string, to: string, type: string, content: string, reply_to?: string, team?: string, timestamp: string }

// Team changed (member joined, left, or team created/deleted)
{ event: "team:changed", team: string, members: string[], action: "joined" | "left" | "created" | "deleted" }
```

### Hub Internal State

All in-memory, no persistence:

```typescript
// Agent registry
Map<string, {
  fullName: string,          // "myproject:andrew@laptop"
  shortName: string,         // "myproject"
  user: string,              // "andrew"
  host: string,              // "laptop"
  ws: WebSocket,             // live connection
  teams: Set<string>,        // team memberships
  connectedAt: Date,
}>

// Disconnected agents (for team membership timeout)
Map<string, {
  fullName: string,
  teams: Set<string>,
  disconnectedAt: Date,      // membership expires at disconnectedAt + 2h
}>

// Teams
Map<string, Set<string>>     // team name → set of agent full names
```

## API / Interface Design

### REST API (Hub)

All endpoints return JSON.

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/` | Dashboard HTML page |

### Setup Endpoint

`GET /setup` returns a shell script:

```bash
#!/bin/bash
HUB="<resolved hub address>"
claude mcp add --transport stdio \
  --env CLAUDE_NET_HUB=http://$HUB \
  claude-net -- bun run http://$HUB/plugin.ts
echo "claude-net registered. Start Claude Code with:"
echo "  claude --dangerously-load-development-channels server:claude-net"
```

Hub address resolution:
1. If `CLAUDE_NET_HOST` env var is set on the hub, use that
2. Otherwise, extract from the `Host` header of the incoming request

## Error Handling

### Plugin Errors

| Scenario | Tool Response |
|----------|--------------|
| Not connected to hub | `"Error: Not connected to hub. Claude Code will auto-connect on next restart, or use register tool."` |
| Name already taken | `"Error: Name 'X' is already registered by another agent. Choose a different name."` |
| Recipient not online | `"Error: Agent 'X' is not online."` |
| Ambiguous short name | `"Error: Multiple agents match 'X': review:andrew@laptop, review:bob@desktop. Use the full name."` |
| Team doesn't exist (send_team) | `"Error: Team 'X' does not exist."` |
| No online team members | `"Error: No online members in team 'X'."` |
| Hub connection lost | Plugin logs to stderr, MCP tools return connection errors. Plugin attempts reconnect with exponential backoff (1s, 2s, 4s, ... 30s max). |
| WebSocket request timeout | If a `requestId` response is not received within 10 seconds, the tool returns a timeout error. |

### Hub Errors

| Scenario | Behavior |
|----------|----------|
| Invalid JSON frame | Send `{event: "error", message: "Invalid frame"}` and ignore |
| Unknown action | Send `{event: "response", requestId, ok: false, error: "Unknown action"}` |
| Agent disconnects | Remove from agent registry, start 2h team membership timer |
| Malformed message fields | Respond with error describing which field is invalid |

## Security Considerations

- **No authentication.** Network visibility is the security boundary. Anyone who can reach the hub's port can register as an agent and send messages.
- **`from` is hub-stamped.** The hub sets the `from` field on all messages based on the sender's registered identity. Agents cannot spoof their identity.
- **Agents should treat inbound messages as peer requests, not trusted instructions.** The plugin instructions string should make this clear to Claude.
- **The dashboard can send messages.** Anyone with browser access to the hub can send messages to agents. This is intentional for LAN use.
- **No TLS by default.** Traffic is plaintext on the LAN. If TLS is needed, use a reverse proxy or run the hub behind tailscale/wireguard.

## Performance Requirements

- LAN-scale: designed for tens of concurrent agents, not hundreds or thousands
- Message delivery latency: sub-100ms on LAN (WebSocket push)
- Hub startup: sub-second
- Plugin startup: limited by `bun run <url>` fetch time (typically <1s on LAN)
- No rate limiting, no message size caps

## Deployment

### Docker

Default port: **4815**

```dockerfile
FROM oven/bun:1
WORKDIR /app
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile --production
COPY src/ ./src/
ENV CLAUDE_NET_PORT=4815
EXPOSE 4815
CMD ["bun", "run", "src/hub/index.ts"]
```

**Production (built-in):**
```bash
docker run -d -p 4815:4815 claude-net
```

**Development (volume-mount):**
```bash
docker run -d -p 4815:4815 -v ./src:/app/src claude-net
```

**With external hostname:**
```bash
docker run -d -p 4815:4815 -e CLAUDE_NET_HOST=mybox.tail1234.ts.net claude-net
```

### Environment Variables

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `CLAUDE_NET_HOST` | no | _(derived from request Host header)_ | External hostname/IP for setup script |
| `CLAUDE_NET_PORT` | no | `4815` | Port the hub listens on |
| `CLAUDE_NET_TLS_CERT` / `CLAUDE_NET_TLS_KEY` | no | _(HTTP)_ | If both set, hub serves HTTPS/WSS; mirror URLs use `https://` |
| `CLAUDE_NET_MIRROR_STORE` | no | _(in-memory)_ | Directory for persistent mirror transcripts. Archives served via `/api/mirror/archive/:sid` |
| `CLAUDE_NET_MIRROR_RETENTION_HOURS` | no | `24` | Archive retention window |
| `CLAUDE_NET_MIRROR_INJECT_RPM` | no | `20` | Per-session inject quota per minute |

### Client Setup

```bash
curl http://<hub-address>:4815/setup | bash
```

Then start Claude Code:
```bash
claude --dangerously-load-development-channels server:claude-net
```
